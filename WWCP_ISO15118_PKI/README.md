# ISO/IEC 15118 V2G Public Key Infrastructure

This C# ISO 15118 PKI builder generates explicit strict or lab V2G certificate hierarchies using BouncyCastle.NET:

- V2G Root → CPO/MO/OEM/CPS
- Sub-CAs (two-tier where appropriate) → SECC / Contract / OEM Prov / CPS Signing leaves
- CRL Distribution Point / AIA metadata and empty CA-issued CRLs

plus an optional set of deliberately evil malformed certificates for pentesting EVCC and SECC implementations.


## Hierarchy

```
V2G Root CA
├── CPO Sub-CA 1 ── CPO Sub-CA 2 ── SECC Leaf            (TLS server, EVSE)
├── MO  Sub-CA 1 ── MO  Sub-CA 2 ── Contract Cert Leaf   (PnC: xmldsig in -2 / mTLS in -20)
├── OEM Sub-CA 1 ── OEM Sub-CA 2 ── OEM Prov Cert Leaf   (factory-installed in EV)
└── CPS Sub-CA   ─────────────────  CPS Signing Leaf     (signs CertificateInstallationRes)
```

## Algorithm profiles

| Profile         | Signing alg     | TLS KEX           | Notes                           |
|-----------------|-----------------|-------------------|---------------------------------|
| `EcdsaP256`     | ECDSA-SHA256    | ECDHE on P-256    | Strict -2 baseline |
| `EcdsaP384`     | ECDSA-SHA384    | ECDHE on P-384    | Lab/interop profile |
| `EcdsaP521`     | ECDSA-SHA512    | ECDHE on P-521    | Strict -20 classical ECC profile |
| `Ed25519`       | Ed25519         | X25519            | Lab/interop profile |
| `Ed448`         | Ed448           | X448              | Strict -20 Edwards-curve profile |
| `MLDsa44`       | ML-DSA-44       | Hybrid/PQC TLS stack required | Experimental PQC, not ISO 15118 compliant |
| `MLDsa65`       | ML-DSA-65       | Hybrid/PQC TLS stack required | Experimental PQC, not ISO 15118 compliant |
| `MLDsa87`       | ML-DSA-87       | Hybrid/PQC TLS stack required | Experimental PQC, not ISO 15118 compliant |

`X25519` and `X448` themselves do not sign certificates; they are TLS-layer key agreement groups paired with EdDSA signatures. Use `--algo 15118-20` to produce the P-521 and Ed448 experimental-strict -20 chains.

### Post-Quantum Cryptography

```bash
dotnet run --project V2GPKI_CLI/V2GPKI_CLI/WWCP_ISO15118_PKI.csproj -- --out ./out --profile pqc --algo pqc --evil
```

This emits three additional experimental chains: ML-DSA-44, ML-DSA-65, and ML-DSA-87. They are useful for sizing, parser hardening, and future interop experiments, but they are not ISO 15118-2/-20 conformance material.

PQC side effects to expect:

- Certificate DER size grows from the sub-kilobyte classical range into multiple kilobytes, depending on ML-DSA level and extensions.
- Signatures are much larger: ML-DSA-44 is already kilobyte-scale, and ML-DSA-87 is larger again.
- Public keys and private keys are much larger, increasing flash use on EVSE/EV devices and making provisioning bundles heavier.
- TLS handshakes, certificate installation messages, and chain transport need larger buffers and more fragmentation tolerance.
- Many TLS libraries, HSMs, secure elements, PKIX validators, and EVCC/SECC stacks will reject PQC OIDs today.
- PQC-only authentication does not solve key agreement; production migration will likely use hybrid classical/PQC TLS and ecosystem-specific policy.
- SLH-DSA/SPHINCS+ is intentionally not generated by default: it has conservative hash-based security but very large signatures, so it is better as an explicit stress-test profile later.

Every generated hierarchy includes `size_report.md` with per-certificate DER, signature, SPKI, and PKCS#8 sizes.

## Build & run

```bash
dotnet build V2GPKI_CLI/WWCP_ISO15118_PKI.slnx
dotnet run --project V2GPKI_CLI/V2GPKI_CLI/WWCP_ISO15118_PKI.csproj -- --out ./out --algo 15118-20 --profile strict-20 --policy-arc 1.3.6.1.4.1.7244.1
```

The refactored layout separates the command-line tool from the reusable builder library:

- `V2GPKI_CLI/V2GPKI_CLI/` contains the `v2gpki` executable.
- `WWCP_ISO15118_PKIBuilder/` contains the hierarchy builder, verifier, I/O, revocation helpers, and evil variant factory.
- `WWCP_ISO15118_PKIBuilderTests/` contains the NUnit regression suite for profile rules and generated artifacts.

Run the tests with:

```bash
dotnet test V2GPKI_CLI/WWCP_ISO15118_PKI.slnx
```

Profiles:

| Profile      | Purpose |
|--------------|---------|
| `strict-2`   | ISO 15118-2 style profile; P-256 only. |
| `strict-20`  | ISO 15118-20 style profile; P-521 and Ed448 only. |
| `lab`        | Interop/pentest profile with broader usages, lab policy OIDs by default, and `--evil` support. |
| `pqc`        | Experimental post-quantum profile for ML-DSA chains. |

Strict profiles do not emit placeholder certificate-policy OIDs. Pass `--policy-arc <oid>` to emit `<oid>.0` root, `<oid>.1` PnC, `<oid>.2` EIM, and `<oid>.3` OEM provisioning policies. Without `--policy-arc`, certificatePolicies are omitted instead of faking standards compliance.

This emits, for each algorithm:

- `out/<profile>_<algo>/01_v2g_root_ca/` … `12_cps_signing_leaf/`, each containing
  `*.cert.pem`, `*.cert.der`, `*.key.pem`, `*.key.der`
- `out/<profile>_<algo>/chains/` with leaf-to-root bundles for each leaf cert
  (`secc_chain.pem`, `contract_chain.pem`, …) plus `v2g_root_trust.pem`
- `out/<profile>_<algo>/crls/` with empty CRLs for every generated CA
- `out/<profile>_<algo>/size_report.md` with certificate/signature/key sizes
- `out/evil_<algo>/<variant>/` with malformed certs and `INDEX.md`

## Evil variants

Generated when `--evil` is passed. Each tests a specific implementation flaw:

| Variant                          | Tests                                                          |
|----------------------------------|----------------------------------------------------------------|
| `expired_secc`                   | NotAfter validation                                            |
| `not_yet_valid_secc`             | NotBefore validation                                           |
| `secc_no_serverauth`             | EKU enforcement on TLS server                                  |
| `contract_with_serverauth`       | EKU privilege escalation: contract cert as SECC                |
| `secc_signed_by_mo`              | Issuer-family policy / certificate-policy OID enforcement      |
| `secc_wrong_san`                 | SAN matching                                                   |
| `path_len_violation`             | basicConstraints pathLenConstraint enforcement                 |
| `leaf_issued_by_leaf`            | cA=false rejection when used as issuer                         |
| `evil_twin_root`                 | DN-collision attack — identical-DN parallel root w/ diff. keys |
| `secc_rsa1024`                   | Weak/non-profile public-key algorithms                         |
| `ed448_leaf_in_ecdsa_chain`      | Mixed-algorithm chain rejection                                |
| `ku_not_critical`                | Validators that only enforce KU when marked critical           |
| `secc_wildcard_san`              | Wildcard SAN policy                                            |

The `evil_twin_root` is particularly worth running end-to-end: it produces a root + sub-CA + SECC leaf with **identical Distinguished Names** to the good hierarchy but completely different keys. An EVCC that pins by DN rather than by SPKI fingerprint will accept it.

The `secc_rsa1024` variant is also worth keeping in the regular pentest set. It creates a SECC certificate whose subject public key is RSA-1024 while the issuer chain remains otherwise valid for the selected profile. This catches validators that only check the certificate path signature math and forget to enforce ISO 15118 algorithm policy, minimum key strength, SPKI algorithm restrictions, and key-usage consistency. In the output it lives under `out/evil_<algo>/secc_rsa1024/`; use the matching good chain as the issuer context when testing path validation.

For example:

```bash
dotnet run --project V2GPKI_CLI/V2GPKI_CLI/WWCP_ISO15118_PKI.csproj -- --out ./out --profile lab --algo p521 --evil
```

## Integration notes

### Plug in to your TLS server (SECC)

For the C# / Norn-style server:

```csharp
var leafChain   = X509CertificateLoader.LoadCertificateFromFile("secc_chain.pem");
var leafKey     = PemEncoding.TryFind…(File.ReadAllText("secc.key.pem"));
var serverCert  = X509Certificate2.CreateFromPemFile("secc.cert.pem", "secc.key.pem");

var sslOptions  = new SslServerAuthenticationOptions {
    ServerCertificate                    = serverCert,
    EnabledSslProtocols                  = SslProtocols.Tls12 | SslProtocols.Tls13,  // -2 + -20
    ClientCertificateRequired            = false,        // -2 default; flip to true for -20 mTLS
    RemoteCertificateValidationCallback  = ValidateContractCert
};
```

For -20 mTLS, point the validation callback at the chain `contract_chain.pem` and pin against the **V2G Root** trust store (`v2g_root_trust.pem`).

### Caveats

1. Strict profiles still need your ecosystem's real CP/CPS values, policy arc, DN rules, validity windows, and publication URLs before production use.
2. Static CRLs are generated and CRL DP / AIA extensions are emitted by default, but no live OCSP responder is started.
3. PQC profiles are experimental. They intentionally sit outside strict ISO 15118-2/-20 mode until the relevant TLS, PKIX, CP/CPS, and ecosystem rules exist.
4. The cert profiles target the common case in ISO 15118-2 Annex H and -20 Annex C; some deployments add national addenda (VDE-AR-E 2802-100-1) that constrain DN ordering, validity periods, or include extra extensions. Add via the `customizer` hook in `V2GCertificateBuilder.Issue`.
